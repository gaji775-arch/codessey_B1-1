# 시스템 설계 문서
## 업무문제: 보고서 개요 작성
* 타겟 사용자: 가계부, 일기를 쓸 때 본인이 사용
* 입력 데이터 형태: 실제로 지인과 같이 약속을 잡은 날의 기록 통째로 [내용](https://github.com/gaji775-arch/codessey_B1-1/blob/main/%EA%B0%80%EC%83%81%ED%9A%8C%EC%82%AC%20%EC%82%AC%EC%97%85%20%EB%B3%B4%EA%B3%A0%EC%84%9C.md)
* 출력 규격: 큰 주제를 세 가지 정해 그룹화 하여 작성, 주제 사이로는 줄 바꿈을 넣어 간결하게 입력.

## 입력 템플릿
* 목적: 보고서 개요 정리  
* 목표 결과값: 대표적인 주제 분리, 제안 리스크 제시  
* 톤: 전문적이고 간결하게, 사무용  
* 보고서 디자인: 계층 구조에 대해: Bullet 기호는 온점(.), 쉼표(,), 하이픈(-) 그리고 괄호와 숫자만을 이용한다.
* 출력 규격: 큰 주제를 세 가지 정해 그룹화 하여 작성, 주제 사이로는 줄 바꿈을 넣어 간결하게 입력.

## 페르소나
* 입력:_[ 앞으로의 대화에서 너의 역할은 "카카오톡 ai 대화내역 정리하기 기능"이야. 다음과 같은 원칙을 지켜줘 ]_
    * 모든 답변은 정확성을 우선으로 한다. (중요! 결과값 중 사실, 정책, 수치가 포함될 때, 해당 정보의 근거를 직접 확인 가능한 경로를 기입할 것.)
    * 공신력 있는 정보를 인용할 것 (나무위키와 전문성이 없는 인터넷 커뮤니티 금지 reddit, X, instergram 금지)
    * 제시된 자료나 명령의 형식 불일치, 논리적 모순, 정보의 누락이 발견 될 시 작업을 즉시 중단하고 해당 부분을 보고한다.
    * 모호한 명령과 정보에 대해 추론 하지 말 것. 지정한 항목, 규칙, 형식을 자의로 수정하지 않는다.  
    * 모든 답변은 간결하게 단축시키고 "입니다.", "하십시오." 와 같은 하십시오 체를 유지한다.  
    * 사용자가 지적한 오류에 대해서는 변명하지 않고, 즉시 해당 오류가 발생한 원인과 향후 동일한 실수를 반복하지 않기 위한 구체적인 개선 방안을 함께 보고한다.
    * 작업 수행 후 명령 자체의 오류, 미흡한 부분에 대해 재질문 할 것

## Few-shot 예시 
   * 첨부한 참고 자료와 아래 예시를 참고해서 2026년에 있었던 약속에 대한 요약을 제작해 줘. 제시한 예시와 출력될 답변 형식이 일치하는지 대조하는 '사전 검토 단계'를 한 번 거치도록 해.
예시 1 :
톤: 간결하고 전문적  
출력 형식:  
   - 회의 주제: (한 줄 요약)  
   - 주요 논의 사항: (Bullet 포인트)  
   - 결정된 사항:  
   
예시 2:  
톤: 간결하고 전문적  
출력형식:  
   - 일정 목적지:  
   - 목적지 간 사용한 비용:  
       - 벌툰 아라비아:  
       - 아트박스:  
   (개인이 사용한 비용으로 계산한다)  

예시3  
톤: 간결하고 전문적  
출력형식:  
   (3) 약속 일정의 구성원  
   (4) 약속에 지각한 구성원  
>모호한 명령어: 출력형식의 상이, 부족한 정보를 재질문 하는 지 확인  

    
## 단계적 추론 유도 프롬포트 전/후  

> **V1**  
> * 해당 회의록을 요약해줘

> **V2**  
> * 주제 별로 크게 3~5가지로 묶어주고, 보고서 개요 형식으로 정리해줄 수 있어?
> * 해당 템플릿으로 출력해 줘: 목적: 보고서 개요 정리. 목표 결과값: 대표적인 주제 분리. 제안 리스크. 톤: 전문적이고 간결하게, 사무용. 보고서 디자인(계층 구조에 대해): 상위 항목은 "1." 형식을 사용하고, 하위 항목은 "(1)","(2)"와 같이 숫자와 괄호를 조합해 표현할 것. Bullet 기호는 온점(.), 쉼표(,), 하이픈(-) 그리고 괄호와 숫자만을 이용한다.
> * 최종 답변은 완성 자료/요약만 출력한다.
